真正值得注意的不是 Dispatch 讓你能在手機遙控桌面，而是它把 agent 的控制面，從「坐下來才會用」推向「隨手可用」的工作流。控制面位置一變，採用邏輯就會被重寫：任務派發更頻繁、回應節奏更碎片化、責任邊界更容易失焦。

Anthropic 這次把 Dispatch 與 computer use 串起來，表面像功能延伸，實際上是工作協調模型的改版。過去我們常問「模型會不會做」；接下來更關鍵的是「任務能不能被穩定地指派、追蹤、接手與審計」。這件事重要，不是因為多一個端，而是因為它把競爭點從模型能力，推向控制面可攜性與治理能力。

## 入口位置改變，先被改寫的是團隊節奏

當任務入口從桌面轉到手機，agent 使用不再綁定完整專注時段，而是滲入會議間隙、通勤途中、跨時區交接前後。這會直接提高任務委派頻率，但也同步提高中途干預與優先序切換的成本。

問題不是能不能做，而是能不能穩定、可觀測、可維運地做。沒有清楚的任務狀態機（例如 queued、running、needs-input、done），手機端只會放大通知噪音，讓團隊以為進度在前進，實際上只是訊息在流動。

## 會擴散的不是最強能力，而是最容易接手的流程

真正會被複製的，通常不是最完整的系統，而是最容易理解、最容易接手的系統。手機控制面讓 agent 從少數專家工具，走向跨角色日常協作；擴散速度因此取決於流程模板化程度，而不是單次 demo 的亮眼結果。

這代表產品與平台團隊要把重心放在「可教學、可複用、可交接」：任務模板、標準輸出格式、最小操作手冊、失敗後的接手路徑。若新成員無法在短時間接手，規模化只會帶來更多半完成任務。

## 競爭點正在移位：從模型表現到治理能力

當手機成為控制面，觸發成本降低，治理缺口會更快暴露：誰可以派工、誰可以改參數、誰可以中斷、誰對結果負責。若這些邊界不明確，組織得到的只是更快的任務發送，而不是更可靠的執行系統。

跨裝置操作也提高了審計要求。沒有端到端事件軌跡，就很難在事故後回答三個基本問題：發生了什麼、誰做了什麼、為什麼當時那樣做。這不只是資安議題，也是交付可信度與管理可辯護性的問題。

## 放回團隊日常，先看二階影響

如果把它放回團隊日常，應該先問的是：在擴大手機委派前，權限邊界、任務狀態可見性與人工接手節點，是否已經定義完成。

二階影響通常會同時出現在六個面向：
- 採用成本：入口變簡單，但跨角色訓練與流程對齊成本上升。
- 可維運性：長任務需要可中斷、可重試、可轉交，否則值班壓力集中在少數人。
- 可觀測性：必須有任務時間線，才能區分是模型失誤、工具故障、權限問題，還是交接失敗。
- 認知負荷：狀態推送過少會失去上下文，過多則造成決策疲勞。
- 責任邊界：產品、平台、資安、法遵之間若無清楚分工，問題會在事故時才被看見。
- 評估負擔：不能只看成功率，還要看人工接手率、平均恢復時間、誤觸發率與回應延遲。

## 採用建議：先治理，再擴量

我會建議三階段推進，而不是一次性全面開放。

第一階段（約 2 週）：只開放低風險任務，先建立角色權限矩陣與最小審計欄位。

第二階段（約 4 週）：補上任務狀態看板與接手機制，明確定義 needs-input 的 SLA，避免任務卡死在灰色地帶。

第三階段（約 4–8 週）：擴展到跨部門流程，固定每週檢視成功率、人工接手率、平均恢復時間、誤觸發率；任一指標惡化，就先收斂範圍再擴張。

## 風險與限制

最常見的風險有四個：權限漂移、假性完成、通知過載、行動端身份與憑證外洩。限制則包括跨平台狀態一致性、弱網路下同步延遲，以及企業法遵要求導致的 rollout 速度下降。

這些都不是新問題，但在手機委派場景下會被放大且提前出現。遠端派工若先於治理落地，結果通常是更快地累積不可追責的任務，而不是更快地交付可靠成果。

把 agent 帶到手機，不等於把自動化帶進組織；只有把控制面與治理一起移動，才算真正的平台升級。
