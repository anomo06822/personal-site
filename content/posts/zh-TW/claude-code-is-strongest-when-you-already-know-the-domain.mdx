這波 Claude Code 的討論，真正值得注意的不是它又更會寫了，而是社群終於把問題從功能炫技移到責任邊界：哪些任務可以可靠託付，哪些決策仍必須由人承擔。當討論焦點變成可託付性，AI coding 才真正進入團隊治理層，而不只是個人工具層。

我想先把結論講白：Claude Code 真正放大的，是已經存在的 domain judgment；它不會憑空替你補出 domain judgment。需求與驗收越清楚，它越像高槓桿執行層；需求越漂、邊界越模糊，它越容易把不確定性放大成 review 稅。

## 先看清第一個誤區：把可生成當成可交付

很多團隊在導入 AI coding 時，第一眼看到的是產量，最後卡住的是驗證。原因不複雜：模型能生成，不等於組織能吸收。真正值得注意的不是生成速度，而是任務是否被定義到可驗證。

在熟悉領域裡，Claude Code 表現好，通常不是因為它突然更懂業務，而是因為團隊早就有高品質上下文：資料結構、錯誤處理慣例、可接受 trade-off、既有測試邊界。這些其實是你們多年累積的判斷資產，工具只是把它轉換得更快。反過來說，當題目本身還沒定義清楚，模型只能用看似合理去補空白，短期快，長期偏。

## 第二個現實：主成本在驗證，不在生成

問題不是能不能做，而是能不能穩定、可觀測、可維運地做。這句話在 AI coding 情境裡，幾乎就是成敗分水嶺。

如果任務有清楚測試基線，review 會是工程流程的一部分；如果任務邊界含糊，review 會退化成高認知負擔的人肉審計。你會看到資深工程師的時間被 diff 與假設檢查吞掉，交付表面上更快，實際上 rework 更高。

所以成效衡量不要只看多寫了多少。至少要同時追：
- `review time`
- `rework ratio`
- `defect escape rate`

沒有這三個維度，任何 productivity 成績單都可能是錯覺。

## 第三個轉移：競爭點從寫碼速度移到判斷供應鏈

當主流工具都能加速實作，差異不再是誰先出第一版，而是誰先把判斷流程做成可複製系統。真正會被複製的，通常不是最完整的系統，而是最容易理解、最容易落地的系統。

這代表競爭點正在從個人 prompt 技巧，移到團隊的判斷供應鏈設計能力：你有沒有任務分級？有沒有 trust contract？有沒有決策紀錄？有沒有 AI-assisted PR 的標準檢查面？沒有這些機制，AI 只會放大個體差異；有這些機制，AI 才會放大組織吞吐。

## 放回交付現場，你會先遇到的二階影響

把 Claude Code 真正放進日常流程後，常見的二階影響通常是：

- adoption cost 前置：先花時間重做任務盤點與流程分層，而不是先買更多席位。
- operability 壓力上升：失敗怎麼回滾、誰有權覆寫、誰對結果負責，都要先定義。
- observability 要求提高：你必須追得出錯誤假設是在哪個上下文切片被引入。
- cognitive load 重新分配：審查負擔可能集中到資深工程師，形成新的瓶頸。
- ownership boundary 更敏感：PM、TL、工程師之間的人機分工若不清，錯誤會被制度放大。
- evaluation burden 擴張：不只看功能對不對，還要看一致性、穩定性與可維運性。

## 採用建議：先縮小成功面，再擴大覆蓋面

如果你要穩定拿到速度紅利，我建議用四步走：

1. 任務分級先行：A 類（高結構、高可驗證）優先導入；B 類（部分不確定）做人機共編；C 類（高不確定、策略性）維持人主導。
2. 建立 trust contract：每類任務明定可接受錯誤型態、必跑測試、必審 diff 區域。
3. 先上最小可用儀表板：追 `lead time`、`review time`、`rework ratio`、`defect escape rate`。
4. 用月度回顧調邊界：把成功樣板沉澱成 checklist 與範例庫，讓新成員可複製。

同時，暫時不要做的一件事很明確：不要在需求尚未收斂的區域追求端到端自動化。這通常只會製造假性進度，最後在整合期一次還債。

Claude Code 不是產品判斷替代品，而是判斷品質放大器。先有可驗證的 domain judgment，速度才是資產；否則，速度只是更快累積的交付債務。
